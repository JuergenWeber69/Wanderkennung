1. Zielsetzung & Abgrenzung

Die Pipeline soll aus einer registrierten Punktwolke eines Gebäudegeschosses automatisiert ein DXF mit fertigen Wandachsen (inkl. Wandstärke, geschlossenes Grundrisspolygon) erzeugen — als direkte Zeichenunterlage bzw. Ausgangsbasis für die Weiterbearbeitung in BricsCAD. Anders als der CloudCompare-RANSAC-Weg (siehe separates Dokument) deckt diese Pipeline die komplette Kette ab: Ebene → Wandkandidat → Wandachse → geschlossener Grundriss.

Abgrenzung

Nicht Ziel dieser ersten Ausbaustufe: automatische Erkennung von Türen/Fenstern als eigene BIM-Objekte, automatische Stockwerkstrennung ohne Vorgabe, oder eine vollständige IFC-Modellerzeugung. Diese Punkte sind mögliche spätere Ausbaustufen, sobald die Grundpipeline stabil läuft.

2. Gesamtarchitektur

Die Pipeline ist als Abfolge unabhängiger, jeweils für sich testbarer Verarbeitungsschritte konzipiert (klassische Verarbeitungskette, keine monolithische Funktion), damit jede Phase separat kalibriert und bei Bedarf ausgetauscht werden kann:

Rohpunktwolke (E57/LAS) │ ▼ [1] Einlesen & Vorverarbeitung ──▶ bereinigte, downgesampelte Wolke │ ▼ [2] Geschosszuschnitt (Höhenband) ──▶ Teilwolke pro Geschoss │ ▼ [3] Normalenschätzung ──▶ Wolke mit konsistenten Normalen │ ▼ [4] Ebenensegmentierung (RANSAC) ──▶ Liste erkannter Ebenen-Patches │ ▼ [5] Wand-Klassifizierung ──▶ gefilterte Liste Wandkandidaten │ ▼ [6] Fragment-Clustering ──▶ zusammengeführte Wandflächen (Öffnungen überbrückt) │ ▼ [7] Wandpaarung & Achsberechnung ──▶ Mittelachsen mit Wandstärke │ ▼ [8] Eckenschluss ──▶ geschlossenes Grundrisspolygon │ ▼ [9] DXF-Export ──▶ Datei für BricsCAD-Import

Warum Python statt Java

Die relevanten Bibliotheken für Punktwolkenverarbeitung (Open3D, PCL-Bindings, Scikit-learn, Shapely) sind im Python-Ökosystem ausgereift und gut dokumentiert. Ein äquivalenter Java-Stack würde bedeuten, diese Bausteine selbst nachzubauen oder auf deutlich weniger gepflegte Bibliotheken auszuweichen — bei gleichem Ziel unnötiger Mehraufwand.

5. Phase 0 — Zieldefinition & Testdaten

P0

Zieldefinition & Testdaten

Geschätzter Aufwand: 1–2 Tage

Bevor die erste Zeile Code entsteht: ein bis zwei reale, „unordentliche" Bestandsaufnahmen (nicht ideale Beispieldaten) als feste Testgrundlage festlegen. Erwartungshaltung an Genauigkeit definieren (z. B. Wandachse auf ±2 cm, Wandstärke auf ±1 cm). Ohne diesen Schritt fehlt später der Maßstab, an dem die Kalibrierung (Phase 11) beurteilt wird.

7. Phase 2 — Dateneinlesen & Vorverarbeitung

P2

Dateneinlesen & Vorverarbeitung

Geschätzter Aufwand: 3–5 Tage

Warum aufwendiger als es klingt: E57/LAS-Dateien großer Bestandsaufnahmen sprengen schnell den Arbeitsspeicher. Es braucht Chunking/Tiling (wolke in Kacheln laden und verarbeiten, ähnlich dem Prinzip, das PointCab bereits nutzt) statt naivem Volleinlesen.

Teilschritte

  • Format-Erkennung und Einlesen (E57 über pye57, LAS über laspy)
  • Umrechnung in ein einheitliches internes Punktformat (XYZ + ggf. Intensität/Farbe)
  • Rauschfilterung (statistischer Ausreißerfilter, analog zum SOR-Filter aus CloudCompare)
  • Downsampling (Voxel-Grid, z. B. 3–5 mm Kantenlänge) zur Reduktion der Punktzahl auf ein handhabbares Maß
import open3d as o3d

pcd = o3d.io.read_point_cloud(pfad)
pcd, _ = pcd.remove_statistical_outlier(nb_neighbors=8, std_ratio=1.2)
pcd = pcd.voxel_down_sample(voxel_size=0.004)  # 4 mm
Achtung

Downsampling-Größe und Filterstärke sind projektabhängig (Scanner-Rauschverhalten unterscheidet sich je nach Gerät und Aufnahmeabstand) — feste Werte im Code sind ein Anti-Pattern, deshalb Auslagerung in die Konfigurationsdatei (siehe Phase 11).

8. Phase 3 — Geschosszuschnitt

P3

Geschosszuschnitt (Höhenband)

Geschätzter Aufwand: 1–2 Tage

Analog zum bestehenden LISP-Tool PWSCHNITT-BATCH wird eine einfache Textdatei im Format Name;Hoehe eingelesen, um pro Geschoss automatisiert das passende Höhenband aus der Gesamtwolke zu extrahieren. Diese Konsistenz zum bestehenden Ladezentrale-Format erleichtert später die Integration (Phase 12).

def slice_by_height(pcd, z_min, z_max):
    pts = np.asarray(pcd.points)
    mask = (pts[:, 2] >= z_min) & (pts[:, 2] <= z_max)
    return pcd.select_by_index(np.where(mask)[0])

9. Phase 4 — Normalenschätzung

P4

Normalenschätzung

Geschätzter Aufwand: 1–2 Tage

Normalenberechnung mit lokalem Suchradius (proportional zur mittleren Punktdichte nach dem Downsampling in Phase 2) und anschließende konsistente Ausrichtung. Fehlerhafte oder inkonsistente Normalen sind die häufigste Ursache für schlechte Ergebnisse in der nachfolgenden Ebenensegmentierung.

pcd.estimate_normals(
    search_param=o3d.geometry.KDTreeSearchParamHybrid(radius=0.02, max_nn=30)
)
pcd.orient_normals_consistent_tangent_plane(k=15)

10. Phase 5 — Ebenensegmentierung

P5

Ebenensegmentierung (RANSAC)

Geschätzter Aufwand: 4–7 Tage (inkl. erster Parameterkalibrierung)

Kernstück der Pipeline. Iteratives RANSAC: größte Ebene finden, zugehörige Punkte entfernen, nächste Ebene finden — bis keine ausreichend große Ebene mehr übrig ist. Open3D bringt segment_plane als Grundbaustein mit; die Iterationslogik (Schleife über mehrere Ebenen) muss selbst ergänzt werden.

def segment_planes(pcd, distance_threshold, min_points, max_planes=60):
    ebenen = []
    rest = pcd
    for _ in range(max_planes):
        if len(rest.points) < min_points:
            break
        modell, inliers = rest.segment_plane(
            distance_threshold=distance_threshold,
            ransac_n=3, num_iterations=2000
        )
        if len(inliers) < min_points:
            break
        ebene = rest.select_by_index(inliers)
        ebenen.append((modell, ebene))
        rest = rest.select_by_index(inliers, invert=True)
    return ebenen, rest
Achtung — hier entsteht der Großteil des Zeitaufwands

distance_threshold und min_points müssen an echten Daten mehrfach nachjustiert werden: zu strenge Werte übersehen echte Wände, zu tolerante Werte verschmelzen Wand mit angrenzenden Möbelflächen. Diese Kalibrierschleife (Parameter ändern → Ergebnis visuell prüfen → erneut testen) ist iterativ und lässt sich nicht vorab exakt terminieren.

11. Phase 6 — Wand-Klassifizierung

P6

Wand-Klassifizierung

Geschätzter Aufwand: 3–5 Tage

Aus der Liste erkannter Ebenen werden Wandkandidaten anhand mehrerer Kriterien gefiltert:

  • Vertikalität: Winkel der Ebenennormale zur Z-Achse nahe 90°
  • Höhenausdehnung: Ebene muss über einen Großteil der Geschosshöhe reichen (unterscheidet Wand von z. B. einer Tischplattenkante)
  • Flächengröße: Mindestausdehnung in der Horizontalen, um kleine Möbelflächen auszuschließen
def ist_wandkandidat(modell, ebene, min_hoehe, min_breite):
    normale = np.array(modell[:3])
    winkel_zur_z = np.degrees(np.arccos(abs(normale[2])))
    bbox = ebene.get_axis_aligned_bounding_box()
    hoehe = bbox.get_extent()[2]
    breite = max(bbox.get_extent()[0], bbox.get_extent()[1])
    return (80 <= winkel_zur_z <= 100) and hoehe >= min_hoehe and breite >= min_breite

12. Phase 7 — Fragment-Clustering

P7

Fragment-Clustering (Öffnungen überbrücken)

Geschätzter Aufwand: 4–6 Tage

Tür- und Fensteröffnungen zerteilen eine durchgehende Wand in mehrere separate RANSAC-Ebenen mit nahezu identischer Ausrichtung und Position. Diese Fragmente müssen anhand von Koplanarität (gleiche Ebenengleichung innerhalb einer Toleranz) und räumlicher Nähe wieder zu einer logischen „Wand" zusammengeführt werden — klassisch mit DBSCAN auf den projizierten 2D-Mittelpunkten der Fragmente.

from sklearn.cluster import DBSCAN

def gruppiere_fragmente(wandkandidaten, eps=0.3, min_samples=1):
    merkmale = [projiziere_auf_ebene(m, e) for m, e in wandkandidaten]
    labels = DBSCAN(eps=eps, min_samples=min_samples).fit_predict(merkmale)
    return labels
Tipp

Als Clustermerkmal nicht nur die Position, sondern auch den Ebenen-Normalenvektor einbeziehen (z. B. gewichtete Distanz aus Position + Winkelabweichung), sonst werden zwei tatsächlich unterschiedliche, aber räumlich nahe Wände fälschlich zusammengeführt.

13. Phase 8 — Wandpaarung & Achsberechnung

P8

Wandpaarung & Achsberechnung

Geschätzter Aufwand: 5–8 Tage

Für jede gruppierte Wand (Phase 7) wird die gegenüberliegende, parallele Fläche gesucht (z. B. Innen- und Außenseite derselben Wand). Aus dem Abstand der beiden Flächen ergibt sich die Wandstärke, die Mittelebene liefert die Wandachse.

  • Parallelitätsprüfung: Winkel zwischen zwei Ebenennormalen unterhalb einer Toleranz (z. B. 5°)
  • Abstandsprüfung: liegt der Ebenenabstand in einem plausiblen Wandstärkenbereich (z. B. 8–40 cm)?
  • Mittelachse: geometrisches Mittel der beiden Ebenengleichungen

Sonderfälle, die hier behandelt werden müssen: einschalige Wände ohne erkennbare Gegenfläche (z. B. nur eine Seite gescannt — dann Standard-Wandstärke aus Konfiguration annehmen), T-Stöße und Eckverbindungen mehrerer Wände.

14. Phase 9 — Eckenschluss & Grundrisspolygon

P9

Eckenschluss & Grundrisspolygon

Geschätzter Aufwand: 5–8 Tage

Die einzelnen Wandachsen (Liniensegmente aus Phase 8) müssen zu einem geschlossenen Grundrisspolygon verschnitten werden: Enden benachbarter Wandsegmente innerhalb eines Suchradius automatisch verlängern oder kappen, damit sie sich exakt treffen. Mit shapely lassen sich Linien verlängern und Schnittpunkte berechnen.

from shapely.geometry import LineString
from shapely.ops import unary_union, polygonize

def schliesse_ecken(liniensegmente, suchradius=0.15):
    verlaengert = [verlaengere_segment(s, suchradius) for s in liniensegmente]
    vereinigt = unary_union(verlaengert)
    return list(polygonize(vereinigt))
Bezug zur bestehenden LISP-Logik

Die Rechtwinkel-Ausrichtung des bereits vorhandenen PWWAND-SNAP-Tools (Snapping auf 0/90/180/270° innerhalb einer Winkeltoleranz) ist konzeptionell dieselbe Idee wie hier — sie lässt sich als zusätzlicher Nachbearbeitungsschritt auf die berechneten Achsen übertragen, um leicht schiefe Bestandswände optional zu begradigen.

15. Phase 10 — DXF-Export

P10

DXF-Export

Geschätzter Aufwand: 2–3 Tage

Export der fertigen Wandachsen (und optional der Wandstärke als parallele Konturlinien) als DXF, mit Layer-Konvention passend zur bestehenden Ladezentrale (z. B. WAND_ACHSEN, analog zum Layer aus PWWAND-SNAP).

import ezdxf

def exportiere_dxf(polygone, pfad, layer="WAND_ACHSEN"):
    doc = ezdxf.new()
    doc.layers.new(name=layer)
    msp = doc.modelspace()
    for poly in polygone:
        punkte = list(poly.exterior.coords)
        msp.add_lwpolyline(punkte, dxfattribs={"layer": layer, "closed": True})
    doc.saveas(pfad)

16. Phase 11 — Kalibrierung an Realdaten

P11

Kalibrierung an Realdaten

Geschätzter Aufwand: 5–10 Tage (parallel zu P5–P9, nicht rein sequenziell)

Alle Schwellwerte (Distanzschwelle, Winkeltoleranzen, Mindestflächen, Cluster-Radien) werden in eine YAML-Konfigurationsdatei pro Gebäudetyp/Scanner ausgelagert. Anhand der in Phase 0 festgelegten Testdaten werden mehrere Durchläufe gefahren, Ergebnis gegen Referenzmaße geprüft, Parameter nachjustiert. Dieser Schritt ist der Hauptgrund, warum die Gesamtentwicklung eher in Wochen als in Tagen zu denken ist — reale Bestandsdaten verhalten sich nie wie synthetische Testdaten.

17. Phase 12 — Integration in bestehenden Workflow

P12

Integration in bestehenden Workflow

Geschätzter Aufwand: 3–5 Tage

Ein batch_runner.py analog zum bestehenden PWSCHNITT-BATCH-Prinzip: liest dieselbe Geschossliste (Name;Hoehe), verarbeitet jede Zeile automatisch durch die komplette Pipeline und legt pro Geschoss ein DXF im Projektverzeichnis ab — bereit für den direkten Import in BricsCAD und die Weiterverarbeitung mit den vorhandenen LISP-Tools.

20. Risiken & Grenzen

  • Stark abweichende Bauzustände (historische Gebäude mit unregelmäßigen, gekrümmten Wänden) sprengen das Grundmodell „ebene Wandfläche" und benötigen Sonderbehandlung
  • Sehr dünn beschossene Bereiche (Verdeckungen im Scan) liefern keine ausreichende Punktdichte für zuverlässige Ebenenerkennung
  • Reine Software-Entwicklung ohne dedizierten Entwickler im Haus verlängert die Kalibrierphasen (P5, P8, P9, P11) erfahrungsgemäß deutlich über die reine Programmierzeit hinaus
  • Ergebnis bleibt eine Rohgeometrie-Unterlage — eine finale Prüfung/Freigabe der erzeugten Wandachsen durch eine Person bleibt in jedem Fall notwendig
